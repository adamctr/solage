\section{Maquettes et enchaînement des écrans}
\label{sec:maquettes}

L'application comporte neuf écrans : connexion, inscription, fil d'actualité, détail d'un
message, profil utilisateur, édition de profil, recherche, administration et page d'erreur 404.
Leur enchaînement est formalisé par la figure~\ref{fig:enchainement}.

\begin{figure}[ht]
\centering
\resizebox{\textwidth}{!}{\input{figures/enchainement}}
\caption{Enchaînement des écrans (storyboard). En vert l'écran protégé par authentification,
en orange l'écran réservé à l'administrateur.}
\label{fig:enchainement}
\end{figure}

J'ai délibérément réutilisé les \textbf{conventions d'interface éprouvées} du microblogging — fil
chronologique, zone de composition en haut, réponses imbriquées, action « j'aime » sous chaque
message — plutôt que d'inventer une interface nouvelle. Ces motifs étant déjà familiers aux
utilisateurs, les reprendre maximise l'\textbf{apprenabilité} et sert les principes ergonomiques
de simplicité et de minimalité des affichages. Ma contribution de conception tient dans
l'\textbf{adaptation} : un périmètre réduit et cohérent, une charte graphique minimaliste propre
à Solage, et l'enchaînement des écrans présenté ci-dessus. Le parcours nominal mène de la
connexion au fil d'actualité, depuis lequel l'utilisateur accède au détail d'un message, aux
profils, à la recherche et — s'il est administrateur — à la modération.

J'ai formalisé ces écrans sous forme de \textbf{maquettes}, réalisées avec \textbf{Penpot} (outil
de maquettage \emph{open source}). Ce sont mes propres écrans — identité visuelle, charte et
libellés propres à \projet{},
sans aucun élément de marque tiers. Chaque écran est décliné en version \textbf{bureau} et
\textbf{mobile}, actant une conception \emph{responsive}. Les trois écrans les plus structurants
sont présentés ici ; les autres figurent en annexe~\ref{ann:maquettes}.

\begin{figure}[H]
\centering
\includegraphics[width=0.82\textwidth]{desktop-connexion.png}
\caption{Maquette de l'écran de connexion (bureau).}
\label{fig:maq-connexion}
\end{figure}

\paragraph{Connexion — focaliser sur une tâche unique.} L'écran de connexion isole sa seule
tâche dans une \textbf{carte centrée} : logo et accroche au-dessus, deux champs étiquetés
explicitement (« Email », « Mot de passe »), et un \textbf{unique appel à l'action} mis en avant
par la couleur d'accent violette. L'absence de tout élément concurrent réduit la charge cognitive
et guide l'œil vers l'action attendue.

\paragraph{Fil et détail — hiérarchie visuelle et constance.} Sur le fil comme sur le détail, la
\textbf{zone de composition} reste en haut, immédiatement accessible. Chaque message suit la même
\textbf{hiérarchie visuelle} : nom de l'auteur en gras, horodatage atténué en gris, contenu en
corps de texte, puis les actions (« répondre », « j'aime ») alignées sous le message. Sur mobile,
une \textbf{barre de navigation inférieure} (accueil, recherche, profil) reste constante d'un
écran à l'autre, et un bouton de retour ramène du détail vers le fil — deux repères de navigation
qui ne bougent jamais.

\paragraph{Choix d'accessibilité (RGAA).} Les maquettes intègrent les repères d'accessibilité
visés par le projet : libellés de formulaire explicites associés à chaque champ, contraste
suffisant entre le texte et le fond, et zones d'action (boutons, icônes) dimensionnées pour le
tactile. Ces choix de conception se traduisent dans le code par des attributs \texttt{alt},
\texttt{aria-label} et des libellés \texttt{<label>} (cf. chapitre~\ref{ch:realisations}).

\section{Conception de la base de données}
\label{sec:bdd}

\subsection{Modèle conceptuel des données (MCD)}

Le modèle conceptuel (figure~\ref{fig:mcd}) recense trois entités principales :
\textsc{Utilisateur}, \textsc{Role} et \textsc{Post}. Les réponses sont modélisées par une
association \emph{réflexive} « répond à » sur \textsc{Post} : une réponse \emph{est} un message
qui en référence un autre. Les « j'aime » et les favoris sont des associations entre
\textsc{Utilisateur} et \textsc{Post}, l'association « aime » portant l'attribut
\texttt{created\_at}.

\begin{figure}[ht]
\centering
\resizebox{0.95\textwidth}{!}{\input{figures/mcd}}
\caption{Modèle conceptuel des données (formalisme Merise).}
\label{fig:mcd}
\end{figure}

\paragraph{Les réponses, sans table dédiée.} Une réponse n'est pas un type d'objet distinct :
c'est un \texttt{post} qui en référence un autre via \texttt{reply\_to} (le message auquel on
répond) et \texttt{reply\_to\_parent} (la racine du fil). Ce choix évite une table \texttt{responses}
séparée qui dupliquerait la structure d'un message, et permet de traiter messages et réponses avec
le même modèle et les mêmes requêtes.

\subsection{Modèle logique / physique (MLD, MPD)}

Le passage au modèle logique transforme les associations en clés étrangères. La règle de
nommage notable : le mot \texttt{user} étant \emph{réservé} en PostgreSQL, les colonnes de clé
étrangère se nomment \texttt{user\_id}. La figure~\ref{fig:mld} présente le schéma relationnel
et ses clés étrangères.

\begin{figure}[ht]
\centering
\resizebox{\textwidth}{!}{\input{figures/mld}}
\caption{Modèle logique des données : tables physiques et clés étrangères (\texttt{\#}).}
\label{fig:mld}
\end{figure}

Le modèle physique est concrétisé par le script de création \file{solage.pg.sql}, chargé une
seule fois à l'initialisation du conteneur PostgreSQL. L'extrait suivant en montre les tables
principales ; le script complet figure en annexe~\ref{ann:sql}.

\begin{lstlisting}[style=sql, caption={Extrait du script de création — \file{solage.pg.sql}}]
CREATE TABLE roles (
    id   SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL
);

CREATE TABLE users (
    id        SERIAL PRIMARY KEY,
    name      VARCHAR(255) NOT NULL,
    email     VARCHAR(255) NOT NULL UNIQUE,
    password  VARCHAR(255) NOT NULL,
    role      INT NULL REFERENCES roles(id) ON DELETE SET NULL,
    image     VARCHAR(255) NULL
);
CREATE INDEX users_role_idx ON users(role);

CREATE TABLE posts (
    id              SERIAL PRIMARY KEY,
    user_id         INT NULL REFERENCES users(id) ON DELETE CASCADE,
    date            TIMESTAMP NOT NULL,
    content         TEXT NULL,
    reply_to        INT NULL,
    reply_to_parent INT NULL,
    image           TEXT NULL
);
CREATE INDEX posts_user_idx ON posts(user_id);

CREATE TABLE likes (
    id         SERIAL PRIMARY KEY,
    user_id    INT NULL REFERENCES users(id) ON DELETE CASCADE,
    post       INT NULL REFERENCES posts(id) ON DELETE CASCADE,
    created_at TIMESTAMP NOT NULL
);
\end{lstlisting}

L'\textbf{intégrité} est garantie par les contraintes : \texttt{NOT NULL},
\texttt{UNIQUE(email)}, clés étrangères avec \texttt{ON DELETE CASCADE} (suppression en cascade
des messages d'un utilisateur supprimé) ou \texttt{SET NULL}. Des \textbf{index} accélèrent les
accès fréquents (clés étrangères, fil d'actualité).

\subsection{Évolution du schéma : migrations idempotentes}

Les évolutions additives ne sont pas appliquées à la main : elles passent par un système de
\textbf{migrations idempotentes} (\file{src/Migrations.php}), qui interroge
\texttt{information\_schema} avant d'agir et peut donc être rejoué sans risque.

\begin{lstlisting}[style=php, caption={Migration idempotente — \file{src/Migrations.php}}]
protected function columnExists(string $table, string $column): bool {
    $sql = "SELECT 1 FROM information_schema.columns
            WHERE table_schema = current_schema()
              AND table_name = :table AND column_name = :column LIMIT 1";
    $stmt = $this->db->prepare($sql);
    $stmt->execute([':table' => $table, ':column' => $column]);
    return $stmt->fetchColumn() !== false;
}

public function migrate(): void {
    $this->addColumnIfNotExists('posts', 'reply_to', 'INT', 'NULL');
    $this->addColumnIfNotExists('posts', 'reply_to_parent', 'INT', 'NULL');
    $this->addColumnIfNotExists('posts', 'image', 'TEXT', 'NULL');
}
\end{lstlisting}

\paragraph{Jeu d'essai et restauration.} Un jeu d'essai reproductible
(\file{seed.sql}) peuple une base de test (utilisateurs, messages, « j'aime », réponses).

\paragraph{Sauvegarde et restauration de la base.} La sauvegarde de la base de production est
automatisée par le script \file{scripts/backup.sh}, planifié en \emph{cron} (une fois par jour).
Il exécute \texttt{pg\_dump} \emph{à l'intérieur} du conteneur PostgreSQL — les identifiants ne
transitent jamais par la ligne de commande, ils sont lus dans l'environnement du conteneur — et
écrit un \emph{dump} compressé et horodaté dans \file{backups/}. L'option
\texttt{-{}-clean -{}-if-exists} rend la restauration rejouable (suppression puis recréation des
objets), un renommage atomique garantit qu'un \emph{dump} partiel n'est jamais conservé sous le
nom final, et les sauvegardes de plus de 14 jours sont purgées (rotation). Les \emph{dumps}
restent hors du dépôt Git (\file{backups/} est dans \file{.gitignore}) : ils contiennent des
données.

La restauration sur une base vierge — exigée par le critère C7 (« la base de test peut être
restaurée en cas d'incident ») — réinjecte le \emph{dump} décompressé dans \texttt{psql} :

\begin{lstlisting}[style=bash, caption={Restauration d'un \emph{dump} compressé dans le conteneur}]
gunzip -c backups/solage-AAAA-MM-JJ-HHMMSS.sql.gz | \
  docker compose -f docker-compose.prod.yml exec -T postgres \
  sh -c 'psql -U "$POSTGRES_USER" -d "$POSTGRES_DB"'
\end{lstlisting}
