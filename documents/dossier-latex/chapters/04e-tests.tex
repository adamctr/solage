\paragraph{Environnement de test.} La compétence C9 est implémentée avec \textbf{PHPUnit~11}
(dépendance de développement Composer). Le \emph{bootstrap} \file{tests/bootstrap.php} réutilise
l'\textbf{autoloader maison} de production (\file{includes/autoload.php}) : les classes de
l'application sont chargées exactement comme par \file{public/index.php}, sans autoloader de test
parallèle. Les tests purs et au \emph{mock} s'exécutent sans base ; les tests d'intégration
visent la \textbf{vraie base PostgreSQL}, chaque test tournant dans une \textbf{transaction
annulée} au \emph{teardown} : la base est jetable, son état identique avant et après la suite.

\paragraph{La testabilité découle du design.} Un composant n'est testable unitairement que si
ses dépendances sont substituables. \code{SessionManager} reçoit son \code{UserModel} \textbf{par
injection} : en test on lui passe un \emph{mock}, donc \code{login()} et \code{isAdmin()} se
vérifient \textbf{sans base}. À l'inverse, \code{UserValidator} instancie son modèle en dur
(\code{new UserModel()}) : non \emph{mockable}, il est testé \textbf{en intégration} contre la
base réelle. Ce contraste est un choix assumé.

\paragraph{Tests de sécurité fonctionnels (CSRF, IDOR).} Certaines protections vivent au niveau
HTTP (lecture de \code{\$\_SERVER}, \code{php://input}, \code{exit}) et ne s'automatisent pas
proprement en PHPUnit ; elles sont validées \textbf{fonctionnellement}, application lancée, la
preuve étant le \textbf{statut HTTP} et la \textbf{ligne de journal} :
\begin{itemize}
  \item \textbf{CSRF} — un \code{POST /login} sans jeton renvoie \code{403} (« Token CSRF
        refusé. ») et journalise \code{csrf.token.rejected} ;
  \item \textbf{IDOR} — connecté en \emph{Bob} (modérateur), une tentative de suppression d'un
        message d'\emph{Alice} renvoie \code{403} et journalise \code{post.delete.forbidden}
        (\code{current\_user\_id=2}, \code{post\_owner\_id=1}).
\end{itemize}
Code des tests, sortie complète et démonstrations figurent en annexe~\ref{ann:tests}.

\section{Préparation et documentation du déploiement}
\label{sec:deploiement}

La mise en production est assurée par \textbf{Dokploy}, une plateforme (PaaS) auto-hébergée sur un
VPS : c'est elle qui fournit le Traefik terminant le HTTPS (Let's Encrypt), redirigeant HTTP vers
HTTPS et posant l'en-tête HSTS. L'application est décrite par une pile dédiée
\file{docker-compose.dokploy.yml} --- variante de \file{docker-compose.prod.yml} \emph{sans}
Traefik, pour ne pas entrer en conflit avec celui de Dokploy (deux Traefik se battant pour les
ports~80/443 seraient une faute de conception). Le service \texttt{migrate} applique les migrations
\emph{avant} le démarrage de l'application, et PostgreSQL n'est exposé que sur le réseau interne.
La pile autonome \file{docker-compose.prod.yml} (Traefik embarqué) reste l'alternative « VPS nu »,
documentée dans \file{DEPLOYMENT.md} ; ses \emph{labels} HSTS sont reproduits ci-dessous.

\begin{lstlisting}[style=yaml, caption={Routage HTTPS et HSTS en production — \file{docker-compose.prod.yml}}]
labels:
  - traefik.http.routers.solage.entrypoints=websecure
  - traefik.http.routers.solage.tls.certresolver=le
  - traefik.http.middlewares.solage-hsts.headers.stsSeconds=31536000
  - traefik.http.middlewares.solage-hsts.headers.stsIncludeSubdomains=true
  - traefik.http.routers.solage.middlewares=solage-hsts
\end{lstlisting}

\paragraph{Procédure documentée.} La procédure complète est consignée dans \file{DEPLOYMENT.md},
à la racine du dépôt. Elle couvre, étape par étape :
\begin{itemize}
  \item les \textbf{pré-requis} : serveur Linux avec Docker, nom de domaine dont l'enregistrement
        DNS~\texttt{A} pointe vers le serveur (indispensable au challenge \texttt{TLS-ALPN-01} de
        Let's Encrypt), ports~80 et~443 ouverts ;
  \item les \textbf{variables d'environnement} (\file{.env}), avec le garde-fou
        \texttt{\$\{VAR:?\}} qui \textbf{fait échouer le démarrage} si une valeur manque ;
  \item les \textbf{étapes} de mise en production : \texttt{git pull} puis
        \texttt{docker compose -f docker-compose.prod.yml up -d -{}-build} ; l'ordonnancement
        (Postgres sain~$\to$ \texttt{migrate}~$\to$ application) est garanti par la pile ;
  \item la \textbf{vérification} (\emph{smoke test}) : \texttt{docker compose ps} et deux
        \texttt{curl} contrôlant la redirection \texttt{308} (HTTP~$\to$~HTTPS) et l'en-tête HSTS ;
  \item le \textbf{retour arrière} (\emph{rollback}) : retour à un \emph{tag} Git et
        reconstruction pour le code ; côté données, des migrations \textbf{additives et
        idempotentes} (sans script \emph{down}) que l'ancien code reste capable de lire ;
  \item la \textbf{sauvegarde / restauration} de la base par \texttt{pg\_dump}, prise avant chaque
        déploiement.
\end{itemize}

\paragraph{Scripts de déploiement.} Le déploiement est \textbf{déclaratif} : les scripts sont les
fichiers \file{docker-compose.yml} (développement) et \file{docker-compose.prod.yml} (production),
complétés par \file{bin/migrate.php} qu'exécute le service \texttt{migrate}. Il n'y a pas de script
impératif \file{deploy.sh} : le \texttt{docker compose up} \emph{est} le déploiement, ce qui le
rend reproductible et versionné avec le code.

terminaison TLS, l'exposition de PostgreSQL et le mode de chargement du code

\paragraph{Limite assumée du déploiement.} Dokploy construit l'image \emph{sur le VPS} à partir du

alors par simple repointage de \emph{tag}, plutôt que par reconstruction depuis Git.

\section{Contribution à la mise en production (DevOps)}
\label{sec:devops}

Plusieurs briques DevOps étaient déjà en place : conteneurisation complète (\texttt{docker
compose}), images reproductibles, migrations \textbf{idempotentes} rejouables sans risque, et
journalisation structurée adaptée à une collecte par l'orchestrateur. Il manquait
l'\textbf{automatisation} : elle est désormais assurée par un pipeline GitHub Actions
d'\textbf{intégration} \emph{et} de \textbf{déploiement} continus, déclenché à chaque \emph{push}.

\paragraph{Le pipeline.} Le dépôt étant hébergé sur GitHub, le serveur d'automatisation est
\textbf{GitHub Actions} (hébergé, gratuit, rien à installer). Le fichier
\file{.github/workflows/ci.yml} décrit \textbf{trois \emph{jobs}} : deux contrôles en parallèle
(\texttt{qualite}, \texttt{image}), puis un déploiement conditionnel (\texttt{deploiement}) :
\begin{itemize}
  \item \texttt{qualite} --- sur \textbf{PHP~8.3} (la version de production) : installe les
        dépendances, vérifie le style \textbf{PSR-12} (PHP\_CodeSniffer), lance l'\textbf{analyse
        statique} (PHPStan), puis exécute la \textbf{suite PHPUnit} (40~tests) contre une base
        PostgreSQL ;
  \item \texttt{image} --- \textbf{construit l'image Docker} (le livrable), pour qu'une image qui
        ne se construit plus soit détectée immédiatement, et non le jour du déploiement ;
  \item \texttt{deploiement} --- sur \texttt{main} uniquement et \textbf{seulement si les deux
        précédents sont verts}, appelle le \emph{webhook} de redéploiement de Dokploy.
\end{itemize}

\begin{lstlisting}[style=yaml, caption={Pipeline d'intégration continue --- extrait de \file{.github/workflows/ci.yml}}]
name: CI
on: [push]

jobs:
  qualite:
    runs-on: ubuntu-latest
    services:
      postgres:                       # base PostgreSQL jetable (auth trust)
        image: postgres:16-alpine
        env:
          POSTGRES_DB: solage
          POSTGRES_USER: solage
          POSTGRES_HOST_AUTH_METHOD: trust
        ports: ["5432:5432"]
        options: >-
          --health-cmd "pg_isready -U solage -d solage"
          --health-interval 5s --health-retries 10
    steps:
      - uses: actions/checkout@v4
      - uses: shivammathur/setup-php@v2
        with:
          php-version: '8.3'          # version de production
          extensions: mbstring, pdo_pgsql
      - run: composer install --no-interaction --no-progress
      - run: php vendor/bin/phpcs --report=full
      - run: php vendor/bin/phpstan analyse --no-progress --memory-limit=512M
      - run: psql -h localhost -U solage -d solage -f solage.pg.sql
      - run: php vendor/bin/phpunit
        env:
          DB_HOST: localhost
          DB_PORT: 5432
          DB_NAME: solage
          DB_USER: solage
          DB_PASSWORD: ""           # trust : ignore par Postgres

  image:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t solage-app .

  deploiement:                          # déploiement continu, main uniquement
    needs: [qualite, image]             # seulement si qualité ET build sont verts
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - run: curl -fsSL -X POST "${{ secrets.DOKPLOY_WEBHOOK }}"  # jeton = secret GitHub
\end{lstlisting}

\paragraph{Une base de test éphémère, sans secret.} Mes tests d'intégration visent une vraie
PostgreSQL (section~\ref{ch:tests}). Le pipeline démarre donc un conteneur \texttt{postgres:16}
déclaré comme \emph{service}, attend qu'il soit sain (\emph{healthcheck}) et y charge le schéma
\file{solage.pg.sql}. La base est \textbf{jetable} et joignable seulement depuis le \emph{runner} :
elle utilise l'authentification \texttt{trust}, donc \textbf{aucun mot de passe n'est codé en dur}
dans le pipeline. Les véritables identifiants vivent dans \file{.env} (hors dépôt) --- même
philosophie « base jetable » que les tests, portée en intégration continue.

\paragraph{Analyse statique introduite avec une baseline.} PHPStan est branché au
\textbf{niveau~1}. Introduit sur du code existant, il a relevé cinq alertes héritées --- dont le
couplage de \code{createUser} au \emph{docroot} (un \code{include} relatif). Je les ai \textbf{gelées dans une
\emph{baseline}} versionnée (\file{phpstan-baseline.neon}) : la CI reste verte, la dette héritée
est \textbf{visible et datée}, et \textbf{tout nouveau code est analysé strictement}. La baseline
dégonfle à chaque correction --- c'est la pratique courante pour introduire l'analyse statique sur
un projet existant.

\paragraph{Interprétation du rapport.} À chaque \emph{push}, le pipeline installe PHP~8.3, vérifie
PSR-12, lance PHPStan, exécute les 40~tests PHPUnit contre une PostgreSQL éphémère, et construit
l'image Docker. La figure~\ref{fig:ci-vert} montre un run \textbf{vert} : les deux \emph{jobs}
aboutissent --- le code respecte PSR-12, l'analyse statique ne relève rien au-delà de la baseline,
les 40~tests passent et le livrable se construit. Un run \textbf{rouge} bloquerait l'intégration et
désignerait l'étape fautive (un test cassé par une régression, une image qui ne se construit plus)
que je corrige en local avant de re-pousser. C'est cette boucle \textbf{automatique} --- et non
l'exécution manuelle des outils --- qui garantit qu'aucune modification non vérifiée n'entre dans
la branche.

\caption{Run d'intégration continue au vert : les contrôles de qualité et la construction du

\paragraph{De l'intégration au déploiement continu.} À chaque \emph{push}, qualité, analyse
statique, tests et construction du livrable sont rejoués automatiquement, en quelques minutes :
c'est l'\textbf{intégration continue}. Sur \texttt{main} et \emph{seulement} si ces contrôles sont
verts, le \emph{job} \texttt{deploiement} appelle le \emph{webhook} de \textbf{Dokploy}, qui
reconstruit et redéploie l'application sur le VPS : c'est le \textbf{déploiement continu}. Un
\texttt{push} sur \texttt{main} qui passe la CI part donc en production sans intervention manuelle,
tandis qu'un \texttt{push} sur une branche de travail exécute les contrôles \emph{sans} déployer
(condition \code{if: github.ref == 'refs/heads/main'}). Le jeton du \emph{webhook} vit dans un
secret GitHub, jamais en clair dans le dépôt. Distinguer ainsi les deux étapes --- vérifier, puis
livrer seulement ce qui est vérifié --- est le cœur de la démarche DevOps.

\section{Veille de sécurité}
\label{sec:veille}

publications de l'ANSSI, les bulletins de sécurité de PHP et les bases de vulnérabilités (CVE).

\paragraph{Le déclic IDOR.} Un article sur la fuite de données ayant touché l'URSSAF (accès aux
données d'autres assurés en modifiant un identifiant dans l'URL — une faille \textbf{IDOR}) a
déclenché un audit de \projet{}. Il a révélé que les routes \texttt{/edituser/\{id\}},

(\texttt{AuthMiddleware}) mais pas la \emph{propriété} de la ressource : un utilisateur
authentifié pouvait modifier ou supprimer le contenu d'autrui en changeant l'identifiant. Le
correctif (contrôle « propriétaire ou administrateur » + \texttt{403} + journalisation) est
décrit en section~\ref{sec:metier}.
